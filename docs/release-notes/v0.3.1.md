Maintenance release. Two silent-failure-mode bug fixes, a per-file syscall cache in the match path, and a stale-doc cleanup.

**No breaking changes.** Upgrade is:

```bash
pip install --upgrade dbxignore   # or: uv pip install --upgrade dbxignore
```

(or download the new binaries) followed by restarting the daemon — `systemctl --user restart dbxignore.service` on Linux; log out / back in or `schtasks /Run /TN dbxignore` on Windows.

## What's fixed

### Atomic `state.write()`

`state.json` is now written via `state.json.tmp` + `os.replace` instead of a direct `write_text()` truncate-then-write. Without this, a SIGKILL or power loss between truncate and write completion could leave a zero-length or partial `state.json`; on next daemon start, the singleton check would treat the corrupt file as "no prior daemon" and let a second daemon launch alongside the first.

Narrow trigger conjunction (hard crash within the few-ms write window **and** user re-runs the daemon before the prior process exits), but the failure mode was silent — two daemons writing markers concurrently, hard to attribute back to corrupt state. `os.replace` is POSIX-atomic on Linux and uses `MoveFileExW(MOVEFILE_REPLACE_EXISTING)` on Windows. `dbxignore uninstall --purge` also cleans up a leaked `state.json.tmp` if one is found.

### Reconcile loop survives generic `OSError` on the read side

`_reconcile_path` previously caught only `FileNotFoundError` and `PermissionError` when reading a marker. Any other `OSError` — e.g. `EIO` on a flaky network drive, or `ENOTSUP` from `getxattr` on a filesystem that doesn't support `user.*` xattrs at all — escaped the per-file try/except and silently killed the per-root sweep worker. The user saw no error in the report; markers just stopped being maintained on that root.

The read-side except is now broadened to a generic `OSError` arm with errno classification, mirroring the existing write-side ENOTSUP/EOPNOTSUPP handling so the contract is uniform across read/write and across platforms.

## What's faster

### `is_dir()` syscall hoisted out of `_rel_path_str`

`RuleCache.match()` and `explain()` now compute `path.is_dir()` once per call instead of once per ancestor `.dropboxignore` file. The directory-only-pattern check (`node_modules/` etc.) was repeating the stat syscall `D` times for the same path when `D` ancestor rule files applied. Hoisting reduces sweep stat-syscalls by `N × (D − 1)` — on a 100k-file tree with one nested ruleset, ~100k fewer `is_dir()` calls per hourly sweep.

## Doc cleanup

- README: dropped a stale paragraph in the "Logging and state" subsection that claimed v0.3 transparently reads v0.2.x state from `~/AppData/Local/dbxignore/` with a WARNING. That auto-migration was actually removed in v0.3.0; the README's top-level `## Upgrading from v0.2.x` section already describes the correct manual upgrade path.
