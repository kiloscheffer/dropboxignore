# v0.3.0 ‚Äî `dbxignore` rename + PyPI publishing

**Status:** Design approved 2026-04-23. Ready for implementation planning.
**Target release:** v0.3.0
**Related backlog items:** #11 (PyPI publishing), #12 (rename)
**Supersedes:** n/a

## Context

`dropboxignore` ‚Äî the PyPI distribution name we'd naturally want ‚Äî is taken by an unrelated 2019 project (Micha≈ÇKarol/dropboxignore). We discovered this during v0.2.x while planning the first PyPI upload. Rather than park the project without a `pip install` path, v0.3.0 renames the tool to `dbxignore` and bundles the first PyPI release under the new name.

The rename scope is Option II from the earlier scoping discussion: **everything the project owns gets renamed**; the Dropbox-owned contracts (marker keys, rule file name, Dropbox client behavior) do not.

## Goals

1. Publish v0.3.0 to PyPI as `dbxignore`, usable via `pip install dbxignore`.
2. Rename every project-owned surface consistently so there's no residual `dropboxignore` referent on the user-visible side.
3. Keep the upgrade path for existing v0.2.x users simple and explicit: a documented manual upgrade, no migration code in v0.3.
4. Keep irreversible release steps gated behind human approval.

## Non-goals

- Rename `.dropboxignore` rule files. They are user content and the name reflects the tool's semantics ("Dropbox ignore rules"). Renaming breaks every existing `.dropboxignore` in the wild.
- Rename the NTFS ADS / xattr key `com.dropbox.ignored`. This is Dropbox's contract with itself ‚Äî renaming makes our markers invisible to the Dropbox client.
- macOS support. Remains on the post-v0.4 roadmap.
- Install-in-place migration code (`_legacy_dropboxignore_state_path()` fallback, systemd-unit auto-rename, etc.). The clean-break upgrade path (see below) obviates the need for any.

## Upgrade path ‚Äî clean break (chosen over forward-compat)

**Existing v0.2.x users upgrade manually:**

```
dropboxignore uninstall --purge   # v0.2.x: removes state, logs, service
pip install dbxignore              # (or uv pip install dbxignore)
dbxignore install                  # registers the new service
```

`.dropboxignore` rule files carry over untouched ‚Äî they're the user's own content in user directories and are never modified by install/uninstall.

**Rationale.** v0.2.1's `uninstall --purge` was broadened to remove `state.json`, `daemon.log*`, the state directory, and the systemd drop-in. That existing cleanup *is* the migration tool ‚Äî running it from v0.2.x leaves the filesystem in exactly the state v0.3.0's fresh `install` expects. A forward-compat legacy-read fallback would add three surfaces (state.py legacy read, systemd unit detection, optional env-var alias) each with its own "remove in v0.4" timer. The clean break trades one README section for zero migration debt.

## Rename scope

### Renamed

| Surface | v0.2.x | v0.3.0 |
|---|---|---|
| PyPI distribution | `dropboxignore` | `dbxignore` |
| Python package directory | `src/dropboxignore/` | `src/dbxignore/` |
| Python imports | `from dropboxignore import ...` | `from dbxignore import ...` |
| CLI console script | `dropboxignore` | `dbxignore` |
| Daemon console script | `dropboxignored` | `dbxignored` |
| Logger name | `dropboxignore.<module>` | `dbxignore.<module>` |
| Env var prefix | `DROPBOXIGNORE_ROOT`, `DROPBOXIGNORE_DEBOUNCE_*` | `DBXIGNORE_ROOT`, `DBXIGNORE_DEBOUNCE_*` |
| State dir | `%LOCALAPPDATA%\dropboxignore\`, `$XDG_STATE_HOME/dropboxignore/` | `%LOCALAPPDATA%\dbxignore\`, `$XDG_STATE_HOME/dbxignore/` |
| systemd unit | `dropboxignore.service` | `dbxignore.service` |
| Windows Task Scheduler name | `dropboxignore` | `dbxignore` |
| PyInstaller outputs | `dropboxignore.exe`, `dropboxignored.exe` | `dbxignore.exe`, `dbxignored.exe` |
| GitHub repo | `kiloscheffer/dropboxignore` | `kiloscheffer/dbxignore` |

### Not renamed

- **`.dropboxignore` rule files** ‚Äî user content.
- **`com.dropbox.ignored` marker keys** (NTFS ADS name, Linux xattr name) ‚Äî Dropbox's contract, not ours.
- **Documentation references to Dropbox the product** ‚Äî the project still operates on Dropbox folders.

## Architecture ‚Äî two-PR delivery

### PR #1 ‚Äî Rename (`feature/v0.3-dbxignore-rename`)

**Scope.** File-system and symbol rename. No behavior change. No release. Merges to `main` with a `[Unreleased]` CHANGELOG entry.

**Order of operations** (matters for clean diffs):

1. Package directory rename via `git mv src/dropboxignore/ src/dbxignore/` as a standalone commit ‚Äî preserves `git log --follow` history.
2. `pyproject.toml` updates:
   - `[project].name` ‚Üí `dbxignore`
   - `[project.scripts]` ‚Üí `dbxignore = "dbxignore.cli:main"`, `dbxignored = "dbxignore.cli:daemon_main"`
   - `[tool.hatch.build.hooks.vcs].version-file` ‚Üí `src/dbxignore/_version.py`
   - `[tool.hatch.build.targets.wheel].packages` ‚Üí `["src/dbxignore"]`
   - `.gitignore` line updated to match.
3. Internal imports ‚Äî grep-and-replace `from dropboxignore` and `import dropboxignore` across code and tests.
4. Logger names ‚Äî grep `getLogger("dropboxignore` and `logger="dropboxignore` across code and tests.
5. Env var prefix ‚Äî grep `DROPBOXIGNORE_ROOT`, `DROPBOXIGNORE_DEBOUNCE_` across code, tests, docs.
6. State + log directory names ‚Äî `state.user_state_dir()` changes its final path component. `daemon._log_dir()` inherits automatically (single source of truth per CLAUDE.md).
7. systemd unit ‚Äî single constant `install/linux_systemd.py:UNIT_NAME`.
8. Windows Task Scheduler ‚Äî corresponding constant in `install/windows_task.py`.
9. PyInstaller spec ‚Äî `pyinstaller/dropboxignore.spec` ‚Üí `pyinstaller/dbxignore.spec`; entry-point paths and output names updated inside.
10. Release workflow ‚Äî `.github/workflows/release.yml` PyInstaller invocation and `files:` list.
11. Docs ‚Äî README (examples, install instructions, badge URLs, repo links), CLAUDE.md (architecture text, gotchas, logger references), `docs/superpowers/specs/*.md` and `docs/superpowers/plans/*.md` (CLI examples).
12. GitHub repo rename via Settings ‚Üí General ‚Üí Rename (performed before merge, not in the diff). Update local `git remote set-url origin git@github.com:kiloscheffer/dbxignore.git`. GitHub auto-redirects cover existing URLs.
13. CHANGELOG ‚Äî `[Unreleased]` section at top of `CHANGELOG.md` with the rename entry. No version heading yet.

**Commit structure** (revertability-aligned per CLAUDE.md):

- `feat!: rename package and CLI to dbxignore` ‚Äî mechanical rename (steps 1‚Äì8).
- `ci!: update release workflow and PyInstaller spec for dbxignore` ‚Äî workflow + spec (steps 9‚Äì10).
- `docs!: update README, CLAUDE.md, and specs for dbxignore rename` ‚Äî docs (step 11).
- `docs(changelog): Unreleased section for v0.3 rename` ‚Äî CHANGELOG only.

Each commit subject carries `!` to signal breaking change per Conventional Commits.

**Validation before merge:**

- `uv run pytest -m "not windows_only"` green locally.
- Linux CI leg green (`pytest -m linux_only` + portable subset).
- Windows CI leg green (`pytest -m windows_only` + portable subset).
- `commit-check` green on every commit in `origin/main..HEAD` per CLAUDE.md pre-flight snippet.
- `gh workflow run release.yml` manual dispatch succeeds end-to-end: wheel builds with `Name: dbxignore`, PyInstaller produces `dbxignore.exe` + `dbxignored.exe`, Publish step correctly skipped (no tag).
- Scratch-venv install: `uv pip install dist/dbxignore-*.whl` into a clean venv, `dbxignore --help` and `dbxignored --help` both work.

### PR #2 ‚Äî PyPI publishing + v0.3.0 release (`feature/v0.3-pypi-publish`)

**Scope.** Add PyPI publish job to `.github/workflows/release.yml`, configure Trusted Publishing + environment gate, finalize CHANGELOG and release notes, tag `v0.3.0`.

**PyPI-side setup (one-time, before PR, completed 2026-04-23):**

- Pre-registered trusted publisher on PyPI: owner `kiloscheffer`, repository `dbxignore`, workflow `release.yml`, environment `pypi`. No placeholder version needed.

**GitHub-side setup (one-time, before PR, completed 2026-04-23):**

- `pypi` environment exists under repo settings with required reviewer set to `kiloscheffer`.
- No additional secrets needed ‚Äî OIDC token issuance is ambient given `id-token: write` permission on the publish job.

**Workflow changes.** Split the existing `build` job's Publish step out so the two destinations (GitHub Releases, PyPI) run as independent downstream jobs, both `needs: build` and both tag-gated. Shape:

```yaml
jobs:
  build:
    runs-on: windows-latest
    # ... existing build steps ...
    # Adds: upload-artifact@v4 at end so downstream jobs can consume dist/

  publish-github:
    needs: build
    if: startsWith(github.ref, 'refs/tags/')
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/download-artifact@v4
        with: { name: dist }
      - uses: softprops/action-gh-release@v2
        with:
          files: |
            dist/*.whl
            dist/*.tar.gz
            dist/dbxignore.exe
            dist/dbxignored.exe
          generate_release_notes: true
          token: ${{ secrets.GH_RELEASE_TOKEN || github.token }}

  publish-pypi:
    needs: build
    if: startsWith(github.ref, 'refs/tags/')
    runs-on: ubuntu-latest
    environment: pypi         # triggers the approval gate
    permissions:
      id-token: write          # OIDC token issuance, job-scoped
    steps:
      - uses: actions/download-artifact@v4
        with: { name: dist }
      - name: Strip PyInstaller outputs
        run: rm -f dist/*.exe
      - uses: pypa/gh-action-pypi-publish@release/v1
        with:
          packages-dir: dist/
```

Key points:

- `id-token: write` scoped to the publish job only (not the build job, which runs user-configurable PyInstaller hooks ‚Äî least privilege at job level).
- No `password:` input to `gh-action-pypi-publish` ‚Äî OIDC handles auth.
- `rm -f dist/*.exe` is explicit even though the action's upload regex already filters non-wheel/sdist files, for clarity of intent.
- `publish-github` and `publish-pypi` both gate on tag ref via `startsWith(github.ref, 'refs/tags/')`, so manual `workflow_dispatch` runs still build but skip both publish paths.

**CHANGELOG.** Move `[Unreleased]` ‚Üí `[0.3.0] ‚Äî YYYY-MM-DD`. Add bottom link `[0.3.0]: https://github.com/kiloscheffer/dbxignore/releases/tag/v0.3.0`.

**Release notes** (`docs/release-notes/v0.3.0.md`) ‚Äî hand-crafted body, applied post-publish via `gh release edit v0.3.0 --notes-file docs/release-notes/v0.3.0.md`. Outline:

- Headline: first PyPI release under the new name.
- **Breaking ‚Äî installation changes.** User-facing subset of the rename table.
- **Upgrading from v0.2.x.** The three-command sequence verbatim, plus note that `.dropboxignore` rule files carry over untouched.
- **Why the rename.** One paragraph: PyPI collision with unrelated 2019 project; short name also clearer.
- **Maintainer changes.** PyPI Trusted Publishing + environment gate added to release workflow.
- Link back to `CHANGELOG.md` for the complete entry.

**Tag + release sequence (after PR #2 merges):**

1. `git checkout main && git pull --ff-only`
2. `git tag v0.3.0 && git push origin v0.3.0`
3. Workflow fires: `build` ‚Üí `publish-github` (runs unconditionally once tag-gated) ‚Üí `publish-pypi` (pauses for `pypi` environment approval).
4. Approve the `pypi` environment deployment via GitHub UI.
5. After PyPI publish completes: `gh release edit v0.3.0 --notes-file docs/release-notes/v0.3.0.md` to override auto-generated notes with the hand-crafted body.

## Testing strategy

### Automated (CI-enforced)

The test suite already exercises imports, logger names, env vars, state paths, install-backend output, markers, and reconcile behavior end-to-end ‚Äî a rename that misses a surface generally produces import failures, `caplog.at_level(...)` mismatches, or path assertions failing. What CI catches automatically:

| Surface | Caught by |
|---|---|
| Python imports | Unit tests fail to import |
| Logger names | `caplog.at_level(..., logger="dbxignore.reconcile")` tests |
| Env var names | `test_discover_honors_env_override` + debouncer env tests |
| State dir path | Daemon integration tests create + read `state.json` |
| systemd unit file contents | `test_linux_systemd_*` assertions |
| Windows Task Scheduler XML | `test_windows_task_*` assertions |
| Reconcile + marker behavior | Existing suite, unchanged semantics |

### Manual (not caught by CI)

| Surface | Check |
|---|---|
| PyInstaller outputs exist with right names | Manual dispatch run of `release.yml` after PR #1 merges |
| CLI entry-point scripts are wired to the new names | `uv pip install dist/dbxignore-*.whl` in scratch venv; run `dbxignore --help`, `dbxignored --help` |
| Wheel metadata has `Name: dbxignore` | `unzip -p dist/*.whl '*.dist-info/METADATA' \| head` |
| README badges + repo links resolve | Preview rendered README; click through |
| Test-file grep for residual `"dropboxignore\."` strings | `grep -rn 'dropboxignore' tests/` ‚Äî should only match intentional references (if any) |

### Two-stage dry-run

- **After PR #1 merges, before PR #2 opens.** `gh workflow run release.yml` validates the build path under the new name: wheel metadata, PyInstaller output names, tag-gate correctness. OIDC is *not* exercised (the `publish-pypi` job doesn't exist yet).
- **After PR #2 merges, before tagging v0.3.0.** `gh workflow run release.yml` runs the full workflow with both publish jobs present. Both publish jobs skip (no tag ref). OIDC is still not exercised (the upload step is tag-gated *and* `pypa/gh-action-pypi-publish` only issues an OIDC token when invoking the upload API). The environment gate *is* the manual OIDC checkpoint on the real tag push.

### Upgrade regression on VPS (required before v0.3.0 tag)

Simulate the documented upgrade end-to-end:

```
pip install dropboxignore==0.2.1
dropboxignore install
# drop a .dropboxignore, let daemon run briefly, verify xattr
dropboxignore uninstall --purge
# verify: systemd unit gone, state dir gone, daemon.log gone
pip uninstall dropboxignore
pip install dbxignore
dbxignore install
# verify daemon starts under new unit name; .dropboxignore rules still apply
```

Any surprise (leftover file, stray systemd reference) ‚Üí update README upgrade section with the workaround before tagging. Validates the "migration = documentation" thesis end-to-end.

## Rollback plan

| Failure mode | Response |
|---|---|
| PR #1 breaks `main` after merge | Revert merge commit. v0.2.1 remains the published GitHub release; no PyPI exposure yet. Fix on new branch. |
| v0.3.0 tag pushed but PyPI publish fails before approval | Decline `pypi` environment deployment. Delete tag + GitHub Release (or draft it). Fix and re-tag. Pre-1.0 tag churn is acceptable. |
| PyPI publish succeeds but wheel is broken | PyPI doesn't allow re-uploading the same version. Yank via PyPI UI (not deletion ‚Äî keeps lockfiles working). Cut v0.3.1 with fix. |
| Upgrade-path bug discovered post-release | Iterate README upgrade section in a docs-only PR. If the bug is in v0.3 code (rare under clean-break), cut v0.3.1. |
| GitHub repo rename breaks something external | GitHub redirects cover URLs. Bookmarks, hardcoded SHAs in third-party tooling, external dashboards are on their owners. Fix forward via release notes if the gap affects anything discoverable. |

**Design property.** The only truly irreversible step is the actual PyPI upload. Everything upstream is revertable (rename PR), droppable (tag), or declinable (environment gate). The gate sits at the exact transition from private-CI to public-index, which is where human judgment has the highest leverage.
